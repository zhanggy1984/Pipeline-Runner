# 流水线编排器

半自动软件工程流水线——输入任务描述，自动完成从设计到审查的全流程。
**仅在设计方案确认、A 级任务、不可自动修复的失败时暂停。全部中文输出。**

## 硬性约束

- **设计阶段始终需要用户确认**——技术方案方向必须由人决策
- **A 级编码任务暂停**——展示方案后等用户确认
- **修复闭环最多 3 轮**——超过后升级给用户决策
- **禁止自动修复**：对外接口变更、安全敏感逻辑、数据模型变更、核心业务逻辑
- **不 `git commit` 或 `git push`** 除非用户明确指示
- **状态文件读写**：启动读 pipeline-state.md，每个阶段门禁通过后写 pipeline-state.md

## 流水线全景

```
/pipeline-runner full "用户登录功能"                # 自然语言描述
/pipeline-runner full docs/prd.md                   # 本地 PRD 文件
/pipeline-runner full https://.../prd.md            # 远程 PRD URL
     │
     ├─ 阶段1: 设计 ─── 主会话执行方案设计流程     [暂停: 确认方案方向+确认详细设计]
     │     输出: .claude/workflow/solution.md + project-context.md
     │
     ├─ 阶段2: 拆分 ─── 子代理(任务拆分器)           [不暂停]
     │     输出: INDEX.md + task-*.md
     │
     ├─ 阶段3: 编码 ─── 子代理(后端/前端编码器) × N  [暂停: A级任务/构建失败]
     │     输出: task-*.done + 编译通过的代码
     │
     ├─ 阶段4: 测试 ─── 子代理(测试大师)             [不暂停]
     │     输出: tests/test-index.md (✅ 最终版)
     │
     ├─ 阶段5: 审查 ─── 子代理(代码+安全+性能) 并行  [暂停: 需人工判断]
     │     ├─ 分类发现 → 自动修复闭环(最多3轮)
     │     └─ 交叉校验: 修复后重新审查，检查副作用
     │     输出: review.done
     │
     └─ 阶段6: 完成 ─── 汇总报告
           输出: pipeline.done
```

## 入口点

| 命令 | 输入类型 | 起始阶段 | 前置条件 |
|------|---------|----------|----------|
| `full "<自然语言描述>"` | 自然语言 | 设计 | 无 |
| `full <本地文件路径>` | 本地 PRD | 设计 | 文件存在 |
| `full <URL>` | 远程 PRD | 设计 | URL 可达 |
| `from-split` | — | 拆分 | solution.md 存在 |
| `from-code` | — | 编码 | solution.md + tasks/ 存在 |
| `from-test` | — | 测试 | task-*.done 全部存在 |
| `from-review` | — | 审查 | tests/test-index.md ✅ 最终版 |
| `status` | — | (只读) | pipeline-state.md |
| `reset` | — | (删除状态) | 无 |

## 状态管理

### 状态文件: `.claude/workflow/pipeline-state.md`

```markdown
# 流水线状态

## 当前位置
- 阶段: 设计
- 状态: 运行中
- 输入类型: 自然语言 / 本地PRD / 远程PRD
- PRD来源: {本地路径 或 URL，自然语言则忽略}
- 任务: {用户输入的任务描述}
- 开始时间: {时间戳}
- 最后更新: {时间戳}

## 设计
- 状态: 已完成
- 方案文件: .claude/workflow/solution.md
- 摘要: {方案摘要}

## 拆分
- 状态: 已完成
- 任务数量: 5
- A 级任务: task-03
- 并行组: 2（后端、前端）

## 编码
- 状态: 进行中
- 已完成: task-00, task-01, task-02
- 进行中: task-03（后端编码子代理）
- 待执行: task-04

## 测试
- 状态: 待执行

## 审查
- 状态: 待执行
- 当前轮次: 0/3

## 完成
- (待执行)
```

### 状态读写规则

1. **启动时**: 读取 pipeline-state.md。存在 → 提取当前阶段和状态，从断点继续。不存在 → 从设计阶段开始。
2. **每个阶段门禁通过后**: 写入 pipeline-state.md（更新阶段和状态）。
3. **暂停时**: 写入暂停状态 + 暂停原因。

---

## 阶段 1: 设计

**主会话内执行**。唯一需要用户交互的阶段。全部中文输出。

### 流程

1. 读取 `skills/design/solution-design.md`，遵循其工作流在当前会话完成方案设计:
   - 步骤1: 清理 `.claude/workflow/`（完整清理）
   - 步骤2: **输入预处理**（判定输入类型并获取 PRD）：
     - 输入以 `http://` 或 `https://` 开头 → **远程 PRD**：
       - 用 WebFetch 拉取完整内容（优先 raw 模式，避免截断）
       - 写入 `.claude/workflow/prd.md`
     - 输入指向本地存在的文件 → **本地 PRD**：
       - 复制到 `.claude/workflow/prd.md`
     - 其他 → **自然语言描述**：
       - 直接作为任务上下文（无 prd.md）
   - 步骤3: 澄清需求——确认范围（后端/前端/全栈）。
     **如果 prd.md 存在，基于 PRD 内容理解需求后再澄清。**
   - 步骤4: 探索现有项目（已有项目），确保 codegraph 索引就绪，将扫描结果写入 `.claude/workflow/project-context.md`
   - 步骤5: 提出 2-3 个宏观方案，**用户选择方向** ← 交互点一
   - 步骤6: 细化设计（数据模型DDL、接口契约、核心流程、测试场景）← **全部中文**
   - 步骤7: 自我挑战 3 个薄弱点（如用户同意）
   - 步骤8: 标记风险（A/B/C）
   - 步骤9: 持久化到 solution.md
2. 验证 solution.md 已写入且包含标题
3. **展示设计方案确认摘要**，等待用户确认 ← 交互点二
4. **用户确认后** → 写 state（阶段=拆分，状态=运行中，记录输入类型+PRD来源），进入阶段 2
5. **用户要求修改** → 回到步骤 6 调整设计

---

## 阶段 2: 拆分

**子代理执行，不暂停。完成后直接进入阶段 3。**

### 流程

1. 验证 solution.md 存在
2. 读取 `skills/planning/task-splitter.md`，提取 `## Sub-Agent Prompt` 章节之后的所有内容
3. 启动子代理（构造 prompt = 任务拆分器 Sub-Agent Prompt + "读取 solution.md，执行工作流，直接写文件"）
4. 等待完成
5. 验证 INDEX.md + task-*.md 已生成
6. 展示摘要（信息性，不阻塞）
7. 写 state（阶段=编码，状态=运行中）
8. **直接进入阶段 3，不暂停**

---

## 阶段 3: 编码

**自动执行所有编码任务。仅 A 级任务和构建失败时暂停。**

### 前置检查

读取 solution.md。全栈且 API 契约不完整 → 输出警告但继续。

### 任务执行循环

```
while 存在可执行任务:
  1. 读 INDEX.md，构建依赖图
  2. 找到所有可执行任务（依赖都已标记完成）
  3. 对每个可执行任务:
     a. 确定技能: .java → 后端编码器, .ts/.tsx/.vue → 前端编码器
     b. 检查风险:
        - A 级 → 暂停: 展示任务内容+设计上下文，等用户确认
        - B/C 级 → 直接启动
     c. 并行检测:
        - 读 INDEX.md 的并行组 + 冲突矩阵
        - 2+ 可执行任务跨不同分支且零文件冲突且无 A 级
          → 全部后台并行启动
        - 同分支 → 串行（等前一个完成标记出现再启动下一个）
     d. 构造子代理 prompt:
        “你是 {后端|前端}编码器 for {task-id}”
        + 任务规格（task-N.md 内容）
        + 设计上下文（solution.md 相关部分: API契约、数据模型、业务规则）
        + 项目上下文（project-context.md）
        + 编码器 Sub-Agent Prompt 章节内容
     e. 启动子代理，等待完成
     f. 检查完成标记，验证构建结果
     g. 构建失败 → 展示错误，询问用户
     h. 更新 state
```

### 合并构建验证

所有并行子代理完成后: `mvn compile -q` + `npm run build`。失败 → 暂停询问用户。

### 门禁: 所有 task-*.done 存在 → 进入阶段 4

---

## 阶段 4: 测试

**子代理执行，不暂停。完成后直接进入阶段 5。**

### 流程

1. 验证所有 task-*.done 存在
2. 读取 `skills/testing/test-master.md`，提取 `## Sub-Agent Prompt` 章节内容
3. 启动子代理（prompt = 测试大师 Sub-Agent Prompt + solution.md 测试场景 + git diff + 项目路径）
4. 等待完成
5. 展示摘要（信息性）
6. 写 state（阶段=审查，状态=运行中）
7. **直接进入阶段 5**

---

## 阶段 5: 审查 + 修复闭环

**并行审查 → 分类发现 → 自动修复 → 交叉校验。仅需人工判断的发现暂停。**

### 步骤 5.1: 收集上下文

读 solution.md + tests/test-index.md，确定审查范围（已有项目→git diff，全新项目→全文件）

### 步骤 5.2: 并行启动审查

三个子代理同时启动（后台运行）。每个 prompt 构造方法: 读取对应 skill 文件，提取 Sub-Agent Prompt 章节内容，拼接设计参考+测试结果+审查范围。

- 代码审查器: solution.md 验收标准+接口契约+业务规则 + 测试覆盖 + 审查范围
- 安全审查器: solution.md 安全约束 + 审查范围
- 性能审查器: solution.md 容量/QPS 目标 + 审查范围

### 步骤 5.3: 收集并分类发现

合并三个审查代理的输出，按规则分类:

**可自动修复**: 缺import、缺@Override/@Transactional、缺空检查、未用import/变量、日志级别、魔法数字、字符串拼接改StringBuilder、React列表缺key、console残留、命名修正、简单数据结构替换

**需人工判断**（优先级高于自动修复）: 致命发现、架构变更、接口契约变更、安全逻辑变更、数据模型变更、业务逻辑修正、跨模块文件变更、对外接口变更、测试断言修正、不确定时

### 步骤 5.4: 自动修复闭环

```
轮次 = 1
while 可自动修复发现非空 且 轮次 ≤ 3:
  1. 按文件分组可自动修复发现
  2. 主会话批量应用修复
  3. mvn compile / npm run build → 失败回滚，标记需人工判断，暂停
  4. mvn test / npm test → 失败且相关则回滚，标记需人工判断，暂停
  5. 重新启动审查: 原来哪个审查者发现的就重新启动哪个（同领域复查）
     三个都有发现则三者全部并行重启
     每个 prompt 附交叉校验上下文
  6. 收集新结果，对比:
     - 同发现被同审查者报告 ≥ 2 次 → 修复无效，标记需人工判断
     - 新发现 → 重新分类
  7. 已被标记需人工判断的不再出现
  8. 剩余可自动修复 → 下一轮
  轮次 += 1

循环耗尽 → 剩余全部升级给用户，暂停
```

### 交叉校验规则

每个审查者只复查自己领域的问题，不跨领域。流水线编排器合并报告后检查结论冲突。

**同发现重复升级规则**: 同发现被同审查者报告 ≥ 2 次 → 立即标记需人工判断（不等 3 轮）

### 门禁: 无致命/高危发现残留 → 写 review.done → 进入阶段 6

---

## 阶段 6: 完成

1. 写 `.claude/workflow/pipeline.done`（含时间戳、任务摘要、各阶段指标）
2. 写 pipeline-state.md（阶段=完成，状态=已完成）
3. 展示汇总报告:

```
## 流水线完成

| 阶段 | 状态 | 产出 |
|------|------|------|
| 设计 | ✓ | solution.md + project-context.md |
| 拆分 | ✓ | 5 个任务，2 个并行组 |
| 编码 | ✓ | 5/5 任务完成，构建通过 |
| 测试 | ✓ | 12/12 测试通过 |
| 审查 | ✓ | 2 个自动修复，0 个遗留问题 |

建议下一步: /commit-generator
```

## 暂停条件汇总

| # | 触发条件 | 阶段 | 恢复方式 |
|---|---------|------|---------|
| 1 | 宏观方案确认 | 设计 | 用户选择方案后继续 |
| 2 | 详细设计确认 | 设计 | 用户确认后进入拆分，或要求修改 |
| 3 | A 级编码任务 | 编码 | 用户确认方案后继续 |
| 4 | 构建失败（自动修复无效） | 编码 | 用户决定修复/跳过/暂停 |
| 5 | 合并构建冲突 | 编码 | 用户解决冲突 |
| 6 | 需人工判断审查发现 | 审查 | 用户决策每个发现 |
| 7 | 自动修复导致回归 | 审查 | 用户分析根因 |
| 8 | 修复闭环耗尽（3轮） | 审查 | 用户决定剩余发现 |

**以下不暂停**: 拆分完成、测试完成、B/C 级编码、可自动修复的审查发现

## 错误恢复

| 错误 | 恢复策略 |
|------|---------|
| 子代理超时 | 重试 1 次，仍超时 → 暂停 |
| 子代理返回空 | 重试 1 次，仍空 → 暂停 |
| 状态文件损坏 | 提示用户 reset 后重试 |
| Git 拉取冲突（编码阶段） | 暂停，列出冲突文件 |
| 依赖 skill 文件不存在 | 降级: 流水线编排器直接执行简化版流程 |
| codegraph 不可用 | 降级: 读取关键配置文件 |

## 降级策略

按优先级排列:
1. 方案设计 skill 不存在 → 引导简化设计（问技术栈 → 最小方案）
2. 任务拆分 skill 不存在 → 简化拆分（按功能拆）
3. 编码器 skill 不存在 → 直接编码（仅简单任务）
4. 测试 skill 不存在 → 跳过测试（警告）
5. 审查 skill 不存在 → 执行简化审查或跳过
6. 任一降级时明确告知用户影响
