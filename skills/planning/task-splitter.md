# 任务拆分器

将方案设计文档拆分为可执行的开发任务。**全自动执行，结果直接写入文件，不暂停等待用户确认。全部中文输出。**

## 输入

1. **优先**: 自动检测 `.claude/workflow/solution.md`
2. 如不存在，提示用户先完成方案设计

## 输出

写入 `.claude/workflow/tasks/`:
```
.claude/workflow/tasks/
├── INDEX.md               # 概览: 依赖图 + 关键路径 + 执行顺序 + 并行组 + 冲突矩阵
├── task-01.md             # 每任务一个文件
├── task-02.md
├── ...
```

每个任务文件格式（纯 markdown，全部中文）:
```markdown
# task-01: 定义数据库表和实体类

## 任务描述
{2-3 句话: 做什么、边界、特殊说明}

## 风险级别
B

## 预估工时
2h

## 涉及文件
- src/main/java/.../entity/XxxEntity.java
- src/main/resources/mapper/XxxMapper.xml

## 依赖任务
无

## 验收标准
1. 数据库建表语句正确
2. 实体映射正确
3. 基本增删改查可执行

## 测试场景
{从方案设计分配的相关测试场景}

## 全局上下文引用
{引用方案设计相关章节，编码子代理无需回翻}
```

INDEX.md 包含:
- 完整任务清单表格
- 依赖图 (Mermaid)
- 关键路径标注
- **并行组**: 任务分组 + 子代理就绪列
- **冲突矩阵**: 跨分支文件冲突检测

## 拆分原则

1. **按功能拆分而非按层拆分**: 每个任务应交付完整功能切片，而非"写所有 Controller"
2. **单任务 ≤ 1 工作日 (8h)** : 超出则继续拆
3. **基础设施优先** : 实体定义 → 数据访问层 → 核心服务 → 控制器 → 测试
4. **测试跟随实现**: 单元测试与任务合并；复杂交互的集成测试独立成任务
5. **风险驱动执行**:
   - A 级: 必须先展示方案，等用户确认后才编码（流水线暂停）
   - B 级: 子代理直接编码，完成后汇报
   - C 级: 子代理直接编码，无需汇报
6. **API 契约优先（全栈时）** : 前后端都涉及时间，API 契约为第一个任务，然后前后端分离为并行分支

## 工作流

### 步骤 0: 扫描项目结构

仅限已有项目。检测到 `pom.xml` / `build.gradle` 时:
- 读取 `pom.xml` 获取模块结构和基础包名
- 读取 `CLAUDE.md` 获取项目约定
- 扫描方案设计引用的模块中的现有文件
- **目标: 任务中的文件路径使用真实项目路径，非假设路径**

### 步骤 1: 理解输入

- 读取 `.claude/workflow/solution.md`
- 识别核心功能和边界
- **自动推断实现范围**: 从方案设计提取范围。方案中提到 UI/页面/组件 → 前端；提到 API/服务/数据库 → 后端；两者都有 → 全栈。不询问用户——直接使用推断结果。
- 标记歧义、矛盾或缺口
- 多拆分方向存在时，选回报率最高的并注明原因

### 步骤 1.5: 检测项目状态，插入 task-00

**目标: 判断项目是否需要骨架初始化。如需要，在其他任务之前插入 task-00。**

#### 1.5.1 从方案设计检查覆盖范围

步骤 1 已确定范围。据此检查:
```
后端  → 当前目录是否存在 pom.xml / build.gradle
前端 → 当前目录是否存在 package.json
```

#### 1.5.2 判断新项目 / 已有项目

- **任一方面是全新项目** (构建文件不存在) → 需要 task-00
- **全部是已有项目** (全部构建文件存在) → 跳过

#### 1.5.3 全新项目的自动处理

- 不询问用户项目路径
- 从方案设计提取技术栈信息
- 如方案设计信息不足，使用默认值: Spring Boot 3.x + Java 17 + MyBatis-Plus
- 在当前目录创建项目骨架定义

#### 1.5.4 生成 task-00

```markdown
# task-00: 项目骨架初始化

## 任务描述
初始化项目基础结构，确保构建通过。
后端: Maven + Spring Boot 3.x + MyBatis-Plus + fastjson2 + validation 骨架
前端: Vite + React/Vue + TypeScript + Ant Design + axios 骨架

## 风险级别
B

## 预估工时
1h

## 涉及文件
- pom.xml / build.gradle
- src/main/java/.../Application.java
- src/main/resources/application.yml
- (前端) package.json, vite.config.ts, src/main.tsx

## 依赖任务
无

## 验收标准
1. mvn compile 通过（后端）
2. npm run dev 启动（前端）
3. pom.xml 包含所需依赖
4. package.json 包含所需依赖
```

#### 1.5.5 更新任务编号

- 原 task-01 → task-02，依此类推
- 所有依赖字段中的编号同步更新

### 步骤 2: 识别依赖关系

排序依据:
1. **API 契约（全栈时）** : 前后端间的契约是首要依赖
2. **数据依赖**: 哪些任务产出的实体/表结构被其他任务消费
3. **API 依赖**: 哪些任务定义的 API 被其他任务调用
4. **基础设施依赖**: 配置变更、MQ 主题创建、数据库建表
5. **外部依赖**: 其他团队、第三方服务

### 步骤 3: 评估风险

按风险分类标准对每个任务标签。自动规则:
- 涉及金额计算 → A 级
- 数据库表结构变更 → A 级
- 分片键变更 → A 级
- 新增/修改消息队列消费者 → A 级
- 共享配置变更 → A 级
- 跨模块接口变更 → A 级
- 新增增删改查端点 → B 级
- 工具类/日志/测试 → B 级

### 步骤 4: 输出文件

**直接写入，不等待用户确认。**

1. **写入任务文件**: 每任务一个 `.md` 文件。

   - `## 并行组`: 并行组表格，包含子代理列:
     ```markdown
     ## 并行组

     | 任务 | 分支 | 依赖 | 子代理 |
     |------|------|------|--------|
     | task-02 | 后端 | task-01 | 是 — 与 task-05 无文件冲突 |
     | task-03 | 后端 | task-02 | 否 — 同分支，顺序执行 |
     | task-05 | 前端 | task-01 | 是 — 与 task-02 无文件冲突 |
     ```
     判断"是": 对比不同分支可执行任务的涉及文件。零文件路径重叠 → 是。同分支任务始终"否"。

   - `## 冲突矩阵` (仅跨分支并行任务):
     ```markdown
     ## 冲突矩阵

     | 任务 | 涉及文件 | 分支 | 编码器 |
     |------|----------|------|--------|
     | task-02 | src/main/java/.../entity/*.java, .../mapper/*.xml | 后端 | backend-coder |
     | task-05 | src/types/*.ts, src/services/*.ts | 前端 | frontend-coder |

     未检测到文件冲突，任务可作为并行子代理执行。
     ```

   - **测试场景分配**: 方案设计有测试场景章节时，按功能将场景分配到对应任务。

2. **输出摘要** (信息性，不阻塞):
   ```
   ## 任务拆分完成

   共 5 个任务，1 个 A 级，2 个并行组。

   并行执行指南:
   - task-02 (后端) 和 task-05 (前端) 可并行执行
   - 后端链: task-02 → task-03 → task-04
   - 前端链: task-05 → task-06 → task-07

   文件已写入 .claude/workflow/tasks/，可自助查看修改。
   ```

3. **标注**: 在 INDEX.md 顶部注明:
   ```
   > 自动生成于 2026-06-11。如需调整任务粒度、依赖关系或文件路径，请直接编辑对应任务文件。
   ```

### 步骤 5: 验证

自检:
- 每个任务是否可独立验证？
- 有无循环依赖？
- A 级任务是否已标注？
- 边界条件有无遗漏？
- **子代理就绪**: 跨分支并行任务的涉及文件零重叠？
  - 常见误判: 两个任务都改 `pom.xml` → 即使改不同段也标记冲突
  - 不确定时标记"否 — 保守"，宁可错过并行也不引发合并冲突

## 流水线集成

此技能被流水线编排器作为第二阶段调用。执行完毕后流水线**直接进入编码阶段，不暂停**。用户可自助查看任务文件进行修改。

## Sub-Agent Prompt

```
你是任务拆分器。执行以下工作流:

1. 读取 .claude/workflow/solution.md 作为主输入。从方案内容中提取范围（后端/前端/全栈）——不询问用户。
2. 扫描项目: 读取 pom.xml/CLAUDE.md 获取模块结构和包名。在任务文件中使用真实路径。
3. 检测全新项目: 构建文件缺失 → 插入 task-00（项目骨架）。从方案设计中提取技术栈。默认: Spring Boot 3.x + Java 17 + MyBatis-Plus。
4. 按功能拆分任务（非按层）。每任务 ≤ 8 小时。基础设施先于业务逻辑。测试与实现任务合并。
5. 识别依赖: 全栈时 API 契约优先，然后数据依赖、API 依赖、基础设施、外部依赖。
6. 评估风险: A 级 = 表结构变更、MQ 消费者、分片键、共享配置、跨模块接口、千万元以上金额。B 级 = 新增增删改查、工具类、日志。C 级 = 格式化、搜索。
7. 直接写入文件（不确认）:
   - INDEX.md（概览 + 依赖图 + 关键路径 + 并行组表 + 冲突矩阵）
   - task-N.md 每任务一个文件
   - 标注: "> 自动生成于 {日期}。如需调整请直接编辑对应任务文件。"
   - 并行组: 按分支分组。跨分支 + 零文件重叠 → 子代理="是"。同分支始终"否"。
   - 冲突矩阵: 跨分支文件路径对比。同一文件不同段也标记冲突。
   - 将方案设计中的测试场景分配到相关任务。
8. 验证: 无循环依赖、A 级已标注、子代理就绪状态二次确认。不确定时标记"否 — 保守"。
9. 报告: 任务数量、A 级清单、并行组、子代理就绪任务。

**全部中文输出**。仅代码标识符（类名、方法名、字段名、表名、文件路径）保持英文。

输出: .claude/workflow/tasks/INDEX.md + task-*.md 文件。
```
