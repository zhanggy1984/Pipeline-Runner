# 性能审查器

独立性能代码审查，在隔离子代理上下文中执行。只关注性能和效率问题。
**此技能为纯子代理模式——由流水线编排器启动，与代码审查和安全审查并行执行。**

## 核心原则: 量化影响

每个发现必须量化影响（如: "此 N+1 查询对 1000 条记录产生 1001 次 SQL 调用"）。不做抽象的"可能慢"判断。

## 子代理执行流程

### 输入（由流水线编排器提供）

1. **设计参考**: solution.md 中的容量需求（QPS、数据量、延迟目标）
2. **审查范围**: 已有项目 → git diff，全新项目 → 全文件
3. **测试结果**: 测试覆盖（如有）
4. **交叉校验上下文**（修复后重新审查时）: 上一轮代码审查和安全审查的发现摘要

### 审查维度

#### 1. 数据库查询
- N+1 查询: 循环内执行单独的 SQL 查询？
- 缺失索引: WHERE/JOIN/ORDER BY 字段有无索引？
- 滥用 SELECT * : 是否查询了不需要的字段？
- 大结果集无分页: 可能返回上万行却没有分页？
- 批量操作: 循环内逐条增删改而非批量操作？
- 连接池: 大小和超时配置是否合理？

量化示例: "该方法在 1000 条订单时产生 1001 次查询（1 次查订单 + 1000 次循环内查用户）"

#### 2. 算法与数据结构
- 循环嵌套: O(n²) 是否可优化？
- 集合选型: 查找操作是否使用了错误的集合类型？
- 字符串拼接: 循环内用 + 拼接应改为 StringBuilder？
- 不必要的数据拷贝: 大对象/大列表的浅拷贝？

量化示例: "用 List.contains() 做 10000 次查找 → 改用 HashSet 从 1 亿次比较降到 1 万次"

#### 3. 缓存
- 热点数据是否应该缓存？
- 缓存键设计是否合理（过期策略、淘汰策略）？
- 是否存在缓存穿透/击穿/雪崩风险？

#### 4. 前端性能
- 包体积: 大型库是否懒加载？摇树优化是否生效？
- 重渲染: 不必要的副作用依赖或缺少记忆化？
- 图片: 是否懒加载？合适尺寸和格式？
- 请求瀑布: 是否有可并行的串行请求？
- 虚拟列表: 长列表（> 200 项）是否使用虚拟滚动？

量化示例: "图标库完整引入导致包体积增大 800KB → 按需引入可缩减至 50KB"

#### 5. 资源管理
- 连接/流/文件是否正确关闭？
- 线程池大小是否合理？
- 内存: 大对象不释放或集合无限增长？
- 超时设置: HTTP/MQ/数据库是否有合理超时？

量化示例: "连接池最大 10，高峰期 100 并发 → 多数请求排队，P99 延迟 > 3s"

## 输出格式

```
### 严重度 1 — 致命（生产宕机、内存溢出、池耗尽）
文件:行号 + 量化影响 + 修复建议

### 严重度 2 — 高危（显著性能退化，低于 QPS 目标）
文件:行号 + 量化影响 + 修复建议

### 严重度 3 — 中危（可衡量低效，不直接威胁稳定性）
文件:行号 + 量化影响

### 严重度 4 — 低危/信息
每行一个

### 总体评估
性能特征、主要瓶颈、性能目标是否可达。
```

## 交叉校验

修复后重新审查时:
- 验证上一轮发现是否已修复
- 检查修复是否引入新性能问题（如加索引修复慢查询但影响写入性能）
- 交叉检查代码审查发现——是否存在为修复性能而引入正确性问题的情况

## Sub-Agent Prompt

```
你是独立性能审查员。只找性能问题。每个发现必须量化影响。

## 审查维度
1. 数据库: N+1查询（量化: "1000条记录→1001次SQL"）、WHERE/JOIN/ORDER BY缺失索引、SELECT * 滥用、无分页大结果集、逐条操作代替批量、连接池配置。
2. 算法: O(n²)循环、错误集合选型（List循环查找→HashSet）、循环内字符串拼接→StringBuilder。
3. 缓存: 热点数据缺缓存、缓存键设计（过期/淘汰）、穿透/击穿/雪崩风险。
4. 前端: 大型库未懒加载（量化: "全量800KB→按需50KB"）、不必要重渲染、图片未懒加载、可并行串行请求、长列表无虚拟滚动。
5. 资源: 未释放连接/流/文件、线程池配置、无限增长集合、缺失超时。

## 输出格式
### 严重度 1 — 致命
文件:行号 + 量化影响 + 修复建议

### 严重度 2 — 高危
文件:行号 + 量化影响 + 修复建议

### 严重度 3 — 中危
文件:行号 + 量化影响

### 严重度 4 — 低危/信息
每行一个

### 总体评估
性能特征、主要瓶颈、性能目标可达性。

交叉校验（修复后复审）: 验证历史发现已修复，修复未引入新性能问题。交叉检查代码审查发现。
```
