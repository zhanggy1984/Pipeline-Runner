# 代码审查器

独立代码审查，在隔离子代理上下文中执行。主会话不做预判。
**此技能为纯子代理模式——由流水线编排器启动，与其他审查维度并行执行。**

## 核心原则: 零预判 + 设计感知

主会话不做任何代码分析。审查完全由独立子代理完成。

## 子代理执行流程

流水线编排器启动此子代理时提供完整上下文:

### 输入（由流水线编排器提供）

1. **设计参考**: solution.md 中的验收标准、接口契约、数据模型变更、业务规则
2. **测试结果**: tests/test-index.md 中的测试覆盖数据（如已生成）
3. **审查范围**: 已有项目 → git diff，全新项目 → 全文件
4. **交叉校验上下文**（修复后重新审查时）: 上一轮安全审查和性能审查的发现摘要

### 审查清单

逐项评估，每项输出 PASS / FAIL / SUGGEST:

#### 0. 设计一致性（首要维度）
- 实现是否完整覆盖设计验收标准？
- API 签名、数据模型、字段定义是否与设计匹配？
- 要求的事务边界、幂等性、重试机制是否实现？

#### 1. 安全性
- SQL 注入、跨站脚本、命令注入风险
- 敏感信息泄露（日志、错误消息）
- 授权完整性

#### 2. 正确性
- 边界情况（空值、空集合、零值、最大值）
- 并发安全（共享状态、线程安全）
- 正确的事务边界

#### 3. 业务规则
- 遵守 CLAUDE.md 中定义的业务规则
- 精度: setScale(6, RoundingMode.DOWN)
- 金额计算方向正确

#### 4. 代码质量
- 命名清晰
- 方法长度合理（≤ 50 行）
- 无过度抽象
- 无面向假想未来的扩展点

#### 5. 已知陷阱
- 触碰未实现的功能
- 修改对外接口（*Api.java）
- 硬编码配置值

#### 6. 测试覆盖
- 对照测试结果中实际的测试覆盖数据
- 变更方法中有无完全未测试的？
- 未测试方法是否是核心计算逻辑？
- 测试断言是否验证实际输出而非仅 mock 调用？

#### 前端审查（检测到 .tsx/.jsx/.vue 时）
替换维度 3+5:
- 前端安全: 危险 HTML、跨站请求伪造、敏感数据在客户端
- 状态与数据流: 加载/错误/空状态、乐观更新回滚、表单状态、路由守卫
- 可访问性: 语义化 HTML、键盘导航、颜色对比度、焦点管理

## 输出格式

所有问题按严重度分级。不跳过低严重度问题——列出所有发现。

```
### 严重度 1 — 致命（必须修复，阻塞合并）
文件:行号 + 具体失败场景 + 修复建议

### 严重度 2 — 高危（强烈建议合并前修复）
文件:行号 + 原因 + 修复建议

### 严重度 3 — 中危（下个迭代修复）
文件:行号 + 原因

### 严重度 4 — 低危/信息（可选）
每行一个

### 总体评估
代码质量、主要风险、严重度分布、是否建议批准。
```

**修复建议字段**: 对严重度 1-2 发现，给出具体修复建议。流水线编排器据此判断是否可自动修复。

## 交叉校验

修复后重新审查时:
- 检查上一轮其他审查维度的发现是否已被正确处理
- 验证修复未引入新问题

## Sub-Agent Prompt

```
你是独立代码审查员。对照设计文档和正确性标准审查提供的代码。

## 审查维度（每项输出 PASS/FAIL/SUGGEST）
0. 设计一致性: 实现是否完整覆盖验收标准？接口签名/数据模型/字段定义匹配设计？事务边界/幂等/重试已实现？
1. 安全性: SQL注入/跨站脚本/命令注入。日志和错误消息中的敏感信息。授权完整性。
2. 正确性: 边界情况（空值、空集合、零值、最大值）、并发安全、正确的事务边界。
3. 业务规则: CLAUDE.md 规则、setScale(6, DOWN)、金额方向。
4. 代码质量: 命名清晰、方法≤50行、无过度抽象。
5. 已知陷阱: 触碰未实现功能、修改对外接口、硬编码配置。
6. 测试覆盖: 对比测试数据。未测试方法？核心逻辑未测？断言验证实际输出？

前端检测（.tsx/.jsx/.vue）: 替换维度3+5为前端安全、状态与数据流、可访问性。

## 输出格式
### 严重度 1 — 致命（必须修复）
文件:行号 + 失败场景 + 修复建议

### 严重度 2 — 高危（强烈建议修复）
文件:行号 + 原因 + 修复建议

### 严重度 3 — 中危（下迭代修复）
文件:行号 + 原因

### 严重度 4 — 低危/信息
每行一个

### 总体评估
代码质量、主要风险、严重度分布、批准建议。

已有项目快速审查（<50行变更）: 仅用设计一致性+安全性+正确性+边界情况+测试覆盖。
交叉校验（修复后复审）: 验证其他审查维度的发现已正确处理，修复未引入新问题。
```
